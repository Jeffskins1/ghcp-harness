Agentic Engineering Basics

Agentic engineering is the practice of using coding agents as active participants in software delivery. The agent can inspect code, propose a plan, edit files, run commands, and carry work through implementation and review.

Once an agent can take actions in a codebase, the team needs a way to shape how it works, what context it sees, which constraints it follows, and what evidence it needs before work is considered complete. That is the job of a harness.

Shared Instructions

Persistent project guidance that tells the agent how the system is structured, which commands are valid, what constraints matter, and which lessons the team has learned from prior failures.

Specs

Written task artifacts that capture intent, acceptance criteria, task breakdown, and open questions so that important context lives in the repo instead of in chat history.

Hooks

Scripts that run at defined moments, such as prompt submission, file edit, command completion, task completion, or Git commit and push. Hooks can warn, record state, or block unsafe actions.

Workflow State

Structured artifacts that record where a task is in the loop, including whether Red was observed, Green was observed, evaluator review is required, and completion evidence is present.

Why Harnesses Exist

Coding agents need harnesses because raw model capability does not create reliable engineering behavior. Teams need a practical way to drive agents toward alignment with local architecture, development policy, test discipline, and definition of done. They also need a way to reduce variance across sessions and across developers.

In an existing application, agents can act without enough architectural context, skip test discipline, spread changes across sensitive boundaries, and declare completion before the work is safe to merge. Without a harness, every session depends more heavily on developer vigilance and repeated prompt scaffolding.

Key point: the harness is a control layer for agent-assisted delivery. Its purpose is to improve alignment, increase determinism, raise quality, and reduce the amount of routine human oversight required.

The Full Loop

The harness is built around two connected loops:

  • Feature loop: establish context, plan the work, activate the first runnable task, and continue until no runnable incomplete tasks remain.
  • Task loop: for each active task, move through Red, Green, and task completion checks before advancing.

The task loop is nested inside the feature loop. After the task loop is exhausted for the active feature, the remaining controls sit outside the feature loop at the repo boundary through commit and push validation.

The goal is to make the agent work through the same engineering cycle every time instead of jumping directly from prompt to implementation.

The harness selects the next runnable task from the implementation plan and records it as the current task in workflow state. At feature start, it picks the first runnable task. After a task completes, it advances to the next runnable task automatically.

Expected output: one active task in state with known dependencies, validation modes, and completion conditions.

Enter the Task Loop

Once a task is active, the agent works through the task-level Red to Green loop. The harness records state at the task level. This loop is repeated once per task, not once per feature.

Expected output: disciplined progress through one task at a time.

For the active task, the agent writes the failing test first. The harness expects a recognized failing test run before implementation is considered valid for that task.

Expected output: evidence that the task has entered Red through an actual failing test for the right reason.

The agent implements the minimum code needed to satisfy the task and reruns the validation command. A recognized passing test run marks Green.

Expected output: passing task tests and a state transition that shows Green was observed after Red.

Verify Completion

The harness checks whether task-level evidence is complete. Depending on the task, that can include Green, a passing full-suite command, and evaluator review.

Expected output: objective completion evidence, not only conversational confidence.

Automatic transition

Advance to the Next Task

After one task completes and passes the release gate, the workflow moves the active pointer to the next runnable task automatically. The task loop then repeats until no runnable incomplete tasks remain.

Expected output: feature progress through a sequence of bounded, evidence-backed tasks without manual state management.

Feature Loop Exhausts

When no runnable incomplete tasks remain, the active feature has no next task to advance to. In the current implementation, this is the effective end of the feature loop in workflow state.

Expected output: all tasks in the feature plan are complete and the active task pointer is cleared.

Commit and Push Validation

After the feature loop is exhausted, repo-level controls still apply. Git hooks enforce commit-message rules, pre-commit checks, and pre-push validation before the change leaves the branch.

Expected output: the change clears repo exit controls before review or merge.

Capture Lessons

Once the feature work is complete, the team records recurring failure modes or important operating lessons back into shared project instructions so later sessions start with that knowledge.

Expected output: the system gets more predictable over time instead of repeating the same mistakes.

Red and Green

Red and Green are the key TDD states in the loop.

Red

Red means the task has an observed failing test. This proves that the work has been made concrete and that the test can detect the missing behavior before implementation begins.

Green

Green means the task has an observed passing test after Red. This proves that the implementation satisfied the testable behavior instead of bypassing the discipline of the loop.

Why this matters: the harness is not only asking developers to prefer TDD. It is trying to record evidence that TDD actually happened at the task level.

Controls and Gates

The loop becomes a harness when controls are attached to it. These controls exist to reduce drift, catch unsafe behavior early, and make task completion depend on evidence rather than on optimism.

Spec Guard

Pushes the agent to work from an active spec and current task definition before implementation begins.

File and Path Guards

Protects sensitive areas of the application from casual or accidental edits and creates friction before risky changes.

TDD State Tracking

Records whether Red and Green have actually been observed for the current task.

Completion Gate

Blocks task completion until the required evidence is present, such as Green, full-suite success, and evaluator review.

Evaluator Review

Adds a second review loop for selected tasks that need higher confidence before they are treated as done.

Lesson Capture

Prevents the harness from being static by writing important operational knowledge back into shared instructions.

Autonomy model: the agent can often run large parts of the task loop semi-autonomously once the task is bounded, but developer involvement is still important at planning, evaluator-triggering decisions, repo exit actions, and final acceptance.

What Principals and Advisors Should Evaluate

Question What to Look For
Does the loop improve alignment? Agents consistently anchor work in project instructions and specs before coding.
Does the loop improve determinism? Different developers can drive the agent through similar tasks with similar evidence and outcomes.
Does the loop improve quality? Tests are written earlier, task completion is evidence-based, and risky changes receive more scrutiny.
Does the loop reduce oversight burden? Reviewers spend less time re-establishing context and catching basic process failures.
Is the loop practical in current application work? Local setup, test commands, and project boundaries are clear enough that the loop can run without constant exceptions.

Leadership Questions Before a Pilot

  • Which application or bounded feature area is stable enough to exercise the full loop?
  • Which test commands are reliable enough to support Red, Green, and final completion checks?
  • Which parts of the application should be protected paths from the start?
  • Which task types should require evaluator review?
  • Who owns the shared instructions, hooks, and lessons after the pilot begins?
  • What evidence will show that the harness is improving outcomes rather than adding ceremony?

Target Outcomes for the Lead and Maintainer Workshop

  • A shared understanding of the full harness loop and its control points
  • Agreement on what Red, Green, completion evidence, and evaluator review mean in practice
  • One candidate pilot application or bounded feature area
  • Named owners for instructions, hooks, workflow state, and policy decisions
  • A short list of pilot risks, expected benefits, and success measures
Recommended posture: principals and advisors should first evaluate whether the full loop is coherent, supportable, and worth piloting in real application work. Broader team onboarding should follow after that judgment is made.

Suggested companion materials for the workshop: an architecture overview, a pilot charter template, a repo maintainer checklist, a realistic example feature spec, and a short live walkthrough of the Red to Green loop.
